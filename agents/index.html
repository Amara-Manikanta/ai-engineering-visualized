AI Agent Architecture

Agents are AI systems that can perceive, reason, act, and observe in a loop — using tools, memory, and sub-agents to complete complex tasks autonomously.

📘
Skills
= KNOWLEDGE (what to do)
🔌
MCP
= ABILITY (connect to world)
👥
Subagents
= DELEGATION (parallel work)
⚡
Hooks
= AUTOMATION (runs outside loop)
Step 1

🔄 The Core Agent Loop

The heart of every AI agent is an infinite loop that cycles between four phases. This is also known as the ReAct pattern (Reason + Act).

🤖 Agent Core LLM ↔ Tools
👁️ PERCEIVE Read input: user message, tool outputs, context files
💭 REASON LLM thinks: what is the goal? What tool do I need next?
⚡ ACT Execute: call a tool, write code, read a file, make an API call
📊 OBSERVE Get the result back. Add it to context. Repeat.
🔁 REPEAT Until task is complete or stopping condition is met
💡 Under Linux Foundation governance, the ReAct pattern is the foundation for most modern agent frameworks including LangGraph, AutoGen, and CrewAI.
Component 1

📘 Skills — The Agent's Knowledge

Skills are reusable instruction modules that teach the agent what to do and how to behave in specific scenarios. They are loaded on-demand, not always in context.

📁 .claude/skills/
📄 deploy/SKILL.md
📄 code-review/SKILL.md
📄 testing/SKILL.md
📄 security-audit/SKILL.md
✅ Reusable instruction modules
✅ Loaded on-demand (progressive disclosure)
✅ Scoped to a task or domain
01
Specificity
Each skill covers one domain. A "deploy" skill knows exactly what commands to run, what to check, what can go wrong.
02
On-Demand Loading
Skills aren't loaded until needed. This saves context window space and keeps the agent focused.
03
Progressive Disclosure
Start with a short description. The agent loads the full instructions only when the skill is triggered.
04
Composable
Skills can reference other skills. A "release" skill might call the "test" + "deploy" + "notify" skills in sequence.
Component 2

👥 Subagents — Delegation & Parallelism

Subagents are independent workers with their own context, model, and permissions. The main agent delegates complex sub-tasks to them and waits for results.

🎯
Orchestrator Agent
Breaks the task, delegates, aggregates
💻
Code Reviewer
Read, Grep, Analyze
Isolated context
🔍
Researcher
Search, Fetch, Summarize
Own model
🚀
Deployer
Bash, SSH, Docker
Own permissions
💡 Key Insight Subagents run in parallel. While one researches, another writes code, another runs tests. This is why complex tasks that would take hours can complete in minutes.
Component 3

⚡ Hooks — Deterministic Automation

Hooks are event-driven scripts that run outside the agent's control. They automate responses to system events — deterministically, without the LLM deciding when to trigger them.

PRE-TOOL
Before Execution
Runs before any tool call. Use for validation, security checks, or logging. Can block the action.
Example: Check if bash command is safe before running
POST-TOOL
After Execution
Runs after a tool completes. Use for post-processing, formatting output, or triggering downstream actions.
Example: Auto-format code after every file write
ON-EDIT
On File Change
Fires whenever a file is modified. Use for running linters, tests, or build systems automatically.
Example: Run ESLint whenever a .ts file changes
ON-NOTIFY
Alerts & Logging
Fires when the agent sends a notification. Use for alerts, Slack messages, or structured logging.
Example: Send Slack message when deploy completes
⚠️
Deterministic — NOT controlled by the LLM. This is the key difference. Hooks are guaranteed to run every time their event fires, regardless of what the agent is thinking. They provide predictability in an otherwise probabilistic system.
Architecture

🏗️ The Agent Extension Stack

All the components stack together in a layered architecture. The bottom layers provide foundation and context; higher layers provide capabilities.

🧩 PLUGINS Extend platform capabilities
📘 SKILLS Teach the agent what to do
🔌 MCP ↔ TOOLS Connect to external world
👥 SUBAGENTS Delegate complex parallel work
⚡ HOOKS Automate responses to events
📝 CLAUDE.md Foundation & Context — always-on project memory

When to use what?

ComponentPurposeAnalogy
Skills Teach agent what to do Training Manual 📖
MCP Connect to external systems USB-C Port 🔌
Subagents Delegate complex work Team Members 👥
Hooks Automate responses to events Tripwires 🪤
CLAUDE.md Always-on project context Sticky Note on Monitor 📌
Plugins Extend platform capabilities App Package 📦
Example

🌍 Real-World Example Walk-Through

Here's how all the pieces come together in a realistic competitive-analysis workflow.

1
CLAUDE.md Loads Project Context
When the agent starts, it automatically reads CLAUDE.md which tells it: "You are a market analyst for Acme Corp. Our main competitors are X, Y, Z."
2
Skill Activates — Competitive Analysis Framework
The user says "analyze our competitors." The agent loads the competitive-analysis/SKILL.md, which provides a structured framework (market share, features, pricing, sentiment).
3
MCP Searches Google Drive
The agent uses the Google Drive MCP server to pull the latest product roadmaps, pitch decks, and customer feedback documents directly.
4
Subagent: Market Researcher Gathers Web Data
A "market-researcher" subagent is spun up with search and fetch tools. It independently crawls competitor websites, reads their blog posts, and summarizes findings.
5
Subagent: Technical Analyst Reviews Repos
In parallel, a "technical-analyst" subagent uses the GitHub MCP server to analyze competitor public repos — tracking commit velocity, star growth, and feature releases.
6
Hook Auto-Formats Output & Runs Linter
When the agent writes the final report, a POST-TOOL hook fires automatically — formatting the markdown, adding table of contents, and running a spell-check. No manual step needed.
