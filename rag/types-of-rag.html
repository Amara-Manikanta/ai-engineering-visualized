Types of RAG Architectures

As RAG matures, basic "Naive RAG" is being replaced by sophisticated architectures to handle complex reasoning, multi-modal data, and self-correction.

21. Evolution of RAG

Naive RAG

The standard pipeline: Embed query → Search DB → Augment Prompt → Generate.

Best for: Simple QA over clean documents.

Advanced RAG

Introduces pre-retrieval (query routing/rewriting) and post-retrieval (reranking using Cross-Encoders).

Best for: Production enterprise systems.

Hybrid RAG

Combines dense vector search (semantic) with sparse keyword search (BM25) to get the best of both worlds.

Best for: Domain-specific jargon (e.g. medical acronyms).

Agentic RAG

An AI agent is given RAG as a tool. It reasons if it needs to search, constructs the query itself, and can search multiple times.

Best for: Complex, multi-part questions.

Graph RAG

Instead of vector chunks, data is stored in a Knowledge Graph (nodes and relationships).

Best for: Tracing complex relationships (e.g., fraud detection, supply chain).

Multi-modal RAG

Retrieving not just text, but images, charts, and audio using multi-modal embedding models (like CLIP).

Best for: Analyzing slide decks, scanned PDFs, or videos.

Corrective RAG (CRAG)

The system evaluates retrieved documents. If they are irrelevant, it automatically triggers a web search to find the correct answer.

Best for: Questions requiring external real-time verification.

Self-RAG

The LLM is trained to generate reflection tokens. It critiques its own retrieved context and generation, deciding whether to retry.

Best for: Highly sensitive applications prioritizing faithfulness.

22. When to Use Each Type

if (needs_to_connect_distant_entities) use_graph_rag();
else if (query_has_specific_product_SKUs) use_hybrid_rag();
else if (question_requires_multi_step_logic) use_agentic_rag();
else if (contains_charts_and_graphs) use_multi_modal_rag();
else use_advanced_rag_with_reranking();
