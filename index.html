What is RAG?

Retrieval-Augmented Generation

RAG is a framework that enhances Large Language Models (LLMs) by connecting them to external knowledge bases. Instead of relying solely on training data, RAG retrieves relevant documents at query-time and feeds them to the LLM as context — resulting in more accurate, up-to-date, and grounded responses.

🧠

Without RAG

LLMs answer from training data only. They can hallucinate, miss recent events, and have no access to private documents.

⚠ Hallucinations possible
→
🚀

With RAG

LLMs retrieve relevant chunks from your knowledge base first, then generate answers grounded in real, current data.

✅ Grounded & accurate

Step 1

Document Input

Start with any document — PDFs, Word files, web pages, or text files. These are your knowledge sources that you want the AI to be able to query and reason over.

Key insight: The quality and structure of your source documents directly impacts the quality of RAG responses.

📝 Indexing — Detailed Notes

📄

1. Document Parsing

Raw documents (PDF, DOCX, HTML, etc.) are loaded and converted into plain text. Libraries like PyMuPDF, pdfplumber, or LangChain loaders handle this. The goal is clean, extractable text.

Tools: PyMuPDF · pdfplumber · unstructured · LangChain
✂️

2. Text Chunking

Long documents are split into smaller overlapping chunks (typically 256–1024 tokens with 20–50 token overlap). This ensures no context is lost at boundaries and each chunk is small enough to be semantically focused.

Strategy: Fixed-size · Sentence · Semantic · Recursive
🔢

3. Embedding Generation

Each chunk is passed through an Embedding Model which converts text into a dense numerical vector (e.g. 1536 dimensions for OpenAI). Semantically similar texts produce numerically similar vectors.

Models: text-embedding-ada-002 · BAAI/bge · Cohere · E5
💾

4. Vector Storage (Milvus)

The embedding vectors (along with the original chunk text as metadata) are stored in a vector database like Milvus. These databases are optimized for fast Approximate Nearest Neighbor (ANN) search.

Options: Milvus · Pinecone · Weaviate · Chroma · FAISS

Step 1

User Asks a Question

A user types a natural language question. This is the query that will drive the entire retrieval process. The system needs to find the most relevant pieces of information from the knowledge base to answer this question.

Key insight: RAG is query-driven — retrieval happens fresh for every user question, in real time.

📝 Retrieval — Detailed Notes

👤

1. User Query

The user submits a question in plain language. Unlike keyword search, RAG understands the intent and meaning of the question — not just matching exact words.

Example:"What are the symptoms of flu?"
🔢

2. Query Encoding

The query is run through the same embedding model used during indexing. This creates a query vector in the same vector space as the document chunks — enabling meaningful comparison.

Critical:Same model for indexing & retrieval!
🎯

3. Semantic Search

The query vector is compared to all stored chunk vectors using cosine similarity or dot product. This finds chunks that are semantically close even if different words are used. Much smarter than keyword matching.

Math:similarity = cos(query_vec, chunk_vec)
📦

4. Top-K Relevant Chunks

Milvus returns the top-K most similar chunks (typically K=3 to 10). These chunks contain the most relevant information from your knowledge base to answer the user's question.

Typical:K=3 to 10 chunks returned

📝 Augmentation — Detailed Notes

📋

1. Context Assembly

Retrieved chunks are concatenated and formatted into a context block. Good RAG implementations add chunk metadata (source, page number) for traceability and citation.

📝

2. Prompt Engineering

A carefully crafted system prompt instructs the LLM to answer only from the provided context. This prevents hallucinations and keeps answers grounded in your data.

🔗

3. Query + Context = Prompt

The final prompt combines: System instructions + Retrieved context + User question. This complete prompt is what gets sent to the LLM in the next stage.

⚖️

4. Context Window Limits

LLMs have a maximum context window (e.g., 128K tokens for GPT-4o). The number of retrieved chunks must fit within this limit along with the system prompt and user query.

Balance:More chunks = more context but more cost

📝 Generation — Detailed Notes

🤖

1. The LLM

Any capable LLM can be used — GPT-4, Claude, Gemini, Llama, or any open-source model. The LLM reads the entire prompt and generates a natural language response token by token.

Models:GPT-4o · Claude 3 · Gemini · Llama 3
✅

2. Grounded Responses

Because the LLM is explicitly told to answer only from the provided context, responses are grounded in your actual data. If the answer isn't in the context, the LLM says "I don't know" instead of hallucinating.

📡

3. Streaming

Modern LLM APIs support streaming — tokens are sent to the UI as they're generated, creating the familiar "typing" effect. This improves perceived response time significantly.

API:stream=True in OpenAI API call
📎

4. Citations & Sources

Advanced RAG systems include source citations in the response — telling users which document and page the answer came from. This builds trust and allows verification.

Pattern:Answer + [Source: doc.pdf, p.12]

Summary

🎓 Key Takeaways

01

RAG = Retrieval + Generation

RAG combines a retrieval system (vector search) with a generative model (LLM). Neither alone is as powerful as both together.

02

Embeddings are the key

Vector embeddings allow semantic understanding — finding meaning, not just matching keywords. The quality of embeddings directly impacts retrieval quality.

03

Chunking strategy matters

How you split documents affects what's retrievable. Too large = noisy context. Too small = missing context. Overlap prevents information loss at boundaries.

04

Grounding prevents hallucination

By instructing the LLM to only use provided context, RAG dramatically reduces hallucination — the LLM admits uncertainty instead of making things up.

05

Scalable to any data

RAG works with PDFs, databases, APIs, web pages — any text source. Vector databases like Milvus scale to billions of vectors with millisecond search.

06

Real-time & always current

Unlike fine-tuning, RAG knowledge can be updated instantly — just re-index new documents. No retraining required. Perfect for dynamic, changing knowledge bases.

LangGraph

🔗 Building Stateful AI Agents

LangGraph is a library for building stateful, multi-actor applications with LLMs, modeled as graphs. Think of it as the orchestration layer for complex AI agent workflows that need cycles, branching, and persistent memory.

🔵
Nodes
Individual processing units — an LLM call, a tool execution, or any Python function. Each node receives state and returns updated state.
➡️
Edges
Connections between nodes. Normal edges always go from A→B. Conditional edges route dynamically based on state (if/else branching).
📦
State
A shared TypedDict object passed through every node. Each node can read and update the state. This is how information flows through the graph.
🔄
Cycles
Unlike basic pipelines, LangGraph supports loops. An agent can think → act → observe → think again in a repeating cycle until it decides to stop.
💾
Checkpointing
Built-in persistence. Save graph state to memory or a database. Resume execution from any checkpoint — perfect for long-running or interrupted tasks.
🎭
Multi-Actor
Multiple agents can run as separate nodes in the same graph. Use for supervisor patterns, debating agents, or parallel task execution.

📄 Minimal LangGraph Agent — Code
agent.py
# 1. Define State
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]

# 2. Define Nodes
def call_llm(state: AgentState):
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state: AgentState):
    last = state["messages"][-1]
    if last.tool_calls: return "tools"
    return END

# 3. Build Graph
graph = StateGraph(AgentState)
graph.add_node("agent", call_llm)
graph.add_node("tools", tool_executor)
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tools", "agent")  # Loop back!

app = graph.compile()

Prompt Engineering

✍️ The Art of Talking to AI

Prompt engineering is the skill of designing inputs to get the best outputs from LLMs. A better prompt = dramatically better results — no training required.

01Zero-Shot Prompting

Directly instruct the model with no examples. Relies entirely on the model's pre-trained knowledge.

❌ Weak
"Translate this."
✅ Strong
"Translate the following English text to French. Preserve the formal tone.

Text: 'The meeting has been rescheduled.'"
💡 Be specific about task, format, audience, and tone.
02Few-Shot Prompting

Provide 2-5 input→output examples before your actual question. The model learns the pattern from your examples.

Example 1: Sentiment("I love it!") → Positive
Example 2: Sentiment("Terrible service") → Negative
Query: Sentiment("It was okay, I guess") → ?
💡 Use 3-5 diverse examples covering edge cases.
03Chain-of-Thought (CoT)

Ask the model to reason step-by-step before giving the final answer. Dramatically improves accuracy on complex tasks.

🤔 Think step by step...
↓
📝 Step 1: Identify known values
↓
📝 Step 2: Apply formula
↓
📝 Step 3: Verify result
↓
✅ Final Answer
💡 Add "Let's think step by step" or "Think before answering".
04Role Prompting

Assign a persona or expert role to the model. It "becomes" that expert and adjusts its tone, depth, and knowledge accordingly.

👨‍⚕️ "You are a senior cardiologist..."
👩‍💻 "You are a 10x Python engineer..."
🎓 "You are a Socratic philosophy professor..."
🔒 "You are a cybersecurity expert..."
💡 Combine role + task + format for best results.
05Tree of Thoughts (ToT)

The model explores multiple reasoning branches simultaneously, evaluates each path, and selects the best one — like a chess player thinking several moves ahead.

Problem
Path A
Dead end ✗
Path B
✅ Best answer
Path C
Dead end ✗
💡 Best for complex problem-solving, puzzles, and planning tasks.
06ReAct Prompting

Interleave Reasoning and Action. The model thinks out loud, takes an action (e.g., search), observes the result, then reasons again. The backbone of modern AI agents.

💭 Thought: I need to find today's stock price
⚡ Action: search("AAPL stock price today")
👁️ Observation: AAPL = $195.32
💭 Thought: Now I can answer the question
✅ Answer: Apple's stock is $195.32
💡 This is what powers LangGraph and tool-using agents.
07Structured Output

Force the model to return output in a specific format — JSON, XML, Markdown tables, or custom schemas. Essential for programmatic use.

"Return ONLY valid JSON:
{
  \"sentiment\": \"positive|negative|neutral\",
  \"confidence\": 0.0-1.0,
  \"reason\": \"one sentence\"
}"
💡 Use with OpenAI's response_format or Pydantic models for reliability.

⚡ Golden Rules of Prompt Engineering

✅ Be specific and unambiguous
✅ Give context and background
✅ Specify output format
✅ Define the audience
✅ Iterate and refine
✅ Use delimiters (""" or XML tags)
❌ Assume the model knows context
❌ Use vague instructions
❌ Ask too many things at once
❌ Ignore system prompts
❌ Forget to validate output
❌ Use negatives (say what TO do)
